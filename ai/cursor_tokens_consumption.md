# Cursor 用量中的四类 Token 与控制 Cache Read

本文整理 Cursor 账号用量里 **Cache Read**、**Cache Write**、**Input**、**Output** 的含义，以及 **Agent 场景下 Cache Read 占比高时的原因与应对**。

---

## 1. 四类用量分别是什么

计费分项与底层模型厂商的 **提示词缓存（prompt caching）** 机制对齐。

| 分项 | 含义 |
|------|------|
| **Input（输入）** | 按**普通输入**计价的那部分 token：本轮新问题、新附上的片段，以及**不能或未**走缓存的上下文。 |
| **Output（输出）** | 模型**生成**的 token（回复、代码等）。单价通常高于 Input。 |
| **Cache Write（缓存写入）** | **首次**把可缓存的长上下文写入提示缓存时计费的 token（建立/更新缓存）。 |
| **Cache Read（缓存读取）** | 后续请求中**命中缓存**、从缓存复用的那段上下文对应的 token 数量；单价通常**明显低于**普通 Input。 |

**小结**：Input / Output 是「新输入 + 模型输出」；Cache Write 是「把可复用长上下文写进缓存」；Cache Read 是「后续轮次从缓存里复用」——用量页拆开显示是为了与各家 API 计费分项一致。

**参考**：[Cursor 模型与定价](https://cursor.com/docs/models-and-pricing)、论坛 [Understanding LLM Token Usage](https://forum.cursor.com/t/understanding-llm-token-usage/120673)。

## 2. 为什么 Cache Read 条数常占大头（尤其用 Agent 时）

现象：单次记录里 **Cache Read** 可占 **60%～70%**，甚至出现**百万级** Cache Read，而 Input、Output、Cache Write 多在万～十万级。

**机制上可以这样理解**：

- 每一轮请求都会带上已积累的**对话、规则、被拉入上下文的文件与工具输出**。
- 其中**稳定、重复的长前缀**会走 prompt cache；后续轮次里这段往往计为 **Cache Read**。
- **Agent** 会**多轮**循环（工具调用、读文件、改代码、再推理……），**前缀随轮次变长**，每一轮上的 Cache Read 都会变长。
- 因此 **总 Cache Read 是多轮叠加的结果**，容易出现「单次记录里 Cache Read 极大、而本轮新增 Input 只有一小截」——**不一定等于主观上「又多读了等量新文字」**。

---

## 3. 条数大 ≠ 费用按同比例涨

- **Cache Read** 与 **Input** 通常是**不同单价**（常见描述为 Cache Read 约为 Input 的一折到几折，以官方定价为准）。
- **「Cache Read 占 token 条数的 60%～70%」不等于「费用也有 60%～70%」**；不少情况下，对账单影响更大的是 **Output** 和**全价 Input**。
- 若 **百万级 Cache Read** 与**实际使用体感**严重不符，可对照单价核算金额；社区曾讨论 [Cache Read 统计异常偏高](https://forum.cursor.com/t/cache-read-bug-using-90-million-tokens-120x-more-than-it-should/120544)，必要时向 Cursor 反馈。

## 4. 如何控制 Cache Read 的 token 条数（Agent 使用注意）

目标：**缩短「每一轮都会被带上的那截重复长上下文」**，而不是否定缓存（缓存往往是在降低等价全价 Input 的成本）。

### 4.1 会话与任务

- **新任务开新对话**，避免无关话题堆在同一条超长线程里。
- 长任务可**分阶段**；阶段结束可**新开 Chat**，只带简短交接说明，减少历史堆叠。
- 同一会话若已多轮试错，可考虑**新开会话 + 保留必要结论**，减少无效历史体积（以当前版本是否保留项目记忆为准，原则仍是少轮次、少冗余）。

### 4.2 控制默认进入上下文的内容

- 使用 **`.cursorignore`**，排除 `node_modules`、构建产物、大资源、生成代码、日志等，减少误把大块内容拉进 Agent 上下文。
- **Cursor Rules** 保持**短而高信号**；大文档改为需要时再 **@ 文件**，避免常驻百科全书式规则。
- **@ 引用要精确**（文件或小范围），避免习惯性质的全仓库级上下文。

### 4.3 Agent 行为

- 任务描述**具体**（路径、模块、期望改动），减少**广撒网式搜索**与**反复读大文件**。
- 长日志、大 diff 尽量**截断或只贴相关片段**。
- 后续轮次尽量**增量说明**（「在上次基础上只改 X」），减少无意义重复拉取（仍取决于实际组 prompt 方式，但有利于缩短对话体积）。

### 4.4 工作区

- 超大仓库或多根目录时，**缩小当前工作区**或拆分仓库，减少可被带入上下文的表面积；与工作区过滤、排除无关目录的思路一致：**Agent 可见范围越小，累积前缀越短**。

## 5. 一页 checklist

- [ ] 理解四类分项：Input / Output / Cache Write / Cache Read。
- [ ] 看到 Cache Read 占比高时，先对照**单价**看费用，不单看条数占比。
- [ ] 用 Agent 时：新任务新会话、`.cursorignore`、精简 Rules、精准 @、具体任务描述、少往对话里堆整文件日志。
- [ ] 异常巨额 Cache Read 且与体感不符时：核算金额并考虑向官方反馈。
